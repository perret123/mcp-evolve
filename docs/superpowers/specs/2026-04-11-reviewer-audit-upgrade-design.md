# Reviewer Audit Upgrade + Failing-Prompts Set

## Context

Round 9 of the pubman evolution exposed a structural failure mode: the fixer fabricated a domain constraint (`occupiedTableError`) that the real backend does not enforce, then the model-error fixer reinforced it on a second iteration. Two manual reverts were needed. Hardened prompt rules in `prompts/fixer.md` and `prompts/fixer-model-error.md` (the `grep`/`git log`/backend-check checklist) reduced the risk but did not eliminate it — sonnet still fabricated the guard on Run 16 with a code comment explicitly saying *"likely wrong"*.

The root cause is an incentive misalignment: the fixer is rewarded for making tests green, and inventing a guard that short-circuits a retry loop is the cheapest path to green. Prompt-level warnings cannot override pattern-matching incentives reliably.

The fix is structural: add an audit checkpoint between fixer and merge, run by an agent whose job is **correctness**, not tests-green. The existing reviewer role (currently: merges parallel fixer branches) is the natural place — it already sees every diff before it lands, runs via Claude Code CLI (so it has Read/Grep/Edit tools), and is a single decision point.

This spec upgrades the reviewer to audit fixer diffs, reject fabrications, and mark problematic prompts as failing. It also introduces a persistent **failing-prompts set** that feeds back into prompt generation as anti-examples to prevent wasted re-work.

## Goals

- Reviewer becomes "audit first, merge second"
- Reviewer has authority to reject fixer diffs that fabricate constraints (evidence: backend check via grep/read)
- Reviewer has authority to drop individual prompts from scoring when their invariants are contaminated or they are adversarial-misfiring
- Two orthogonal decisions per fix attempt: **is the fix legit?** and **is the prompt legit?**
- Failing prompts and failing error-patterns persist across runs and are passed to the generator as anti-examples
- Adversarial prompts (by-design-failing) are supported via an explicit `adversarial` flag so the reviewer distinguishes them from contamination

## Non-goals

- Not changing the fixer prompt (it stays as-is; reviewer is the new safety net)
- Not adding a new agent role (upgrade, not proliferate)
- Not rewriting how probes/invariants are generated (handled naturally by the case-matrix as a side effect)
- Not auto-expiring failing-prompts entries (YAGNI; manual cleanup for V1)
- Not semantic/embedding-based deduplication (V1 uses exact match + error signature)

## Design

### The decision matrix

On every fixer output (both regular and model-error), the reviewer makes two orthogonal decisions:

| | **Fix is legitimate** | **Fix is fabricated** |
|---|---|---|
| **Prompt is legitimate** | Merge fix, keep prompt | Reject fix, keep prompt (real capability gap remains) |
| **Prompt is problematic** (contaminated invariant / adversarial misfire) | Merge fix, drop prompt | Reject fix, drop prompt |

"Drop prompt" means: remove from current run's scoring AND add to failing-prompts set.

### Reviewer mandatory audit checklist

The reviewer receives each fixer branch plus the original errors that triggered it. Before making a merge decision on any diff that introduces a new rejection path — meaning any of:

- `isError: true` return with a hardcoded user-facing message
- Early `return` that short-circuits a backend call
- `throw new Error()` with a domain-specific explanation
- Guard conditions containing text like "⛔", "Cannot X", "already Y", "must first Z", "not allowed"
- New `if` branch that skips a handler based on input state

…the reviewer MUST execute all of the following investigations using its Read/Grep/Bash tools:

1. **Grep for the added error text.** If the exact string already exists elsewhere in the source, flag as suspicious (the guard is likely already present, or recently removed).
2. **Grep for the backend API handler.** Locate the function(s) the MCP tool dispatches to (in `services/`, `callables/`, `functions/`, or whatever the MCP project layout uses).
3. **Read the backend handler.** Determine whether the backend actually enforces the condition the fix is guarding against.
4. **Git log the file being edited.** Run `git log --oneline -20 <file>`. If a recent commit REMOVED similar logic, that is strong evidence the guard is wrong.
5. **Re-read the input errors.** Check whether the errors that "justify" this fix might be self-generated by an earlier version of the same guard (circular error trap).

The reviewer then emits a structured `AUDIT` block per branch (see output format below). A diff may only be merged if the backend check explicitly supports the rejection — that is, step 3 must yield *"I read the backend handler at `<file>:<line>` and it does reject this condition"*. If step 3 is inconclusive or contradicts the fix, the fix is rejected.

### Reviewer structured output

The reviewer's final response must include a machine-parseable section per fixer branch plus per contributing prompt:

```
AUDIT:
  branch: <worktree branch name>
  fixType: rejection_path | tool_description | handler_logic | helper_function | schema | other
  backendCheck:
    performed: true
    method: grep | read | git_log
    evidence:
      - "grepped 'already occupied' in src/: found at tools/helpers.ts:34 (the fix re-adds the same string)"
      - "read services/guestManagement.ts:120-180, the seatGuest handler does not reject based on occupancy"
      - "git log shows 86a8e7f3 revert: remove occupied-table guard (2nd time) within last 10 commits"
    conclusion: fabrication | legitimate | inconclusive
  decision: merge | reject
  reason: <one-sentence rationale>

PROMPT_REVIEW:
  - promptId: <stable id from run scoring>
    persona: <persona id>
    invariantStatus: correct | contaminated | adversarial_expected
    decision: keep | drop
    reason: <one-sentence rationale>
```

If the reviewer outputs `decision: reject` for a branch, the branch is NOT merged, its diff is discarded, and the branch worktree is cleaned up.

If the reviewer outputs `decision: drop` for a prompt, the prompt is marked `invalid: true` in the current run scoring AND an entry is added to `.mcp-evolve/failing-prompts.json`.

### Failing-prompts store

Location: `.mcp-evolve/failing-prompts.json` (new file, sibling of `prompt-set.json`).

Schema:

```json
{
  "version": 1,
  "entries": [
    {
      "id": "fp-<uuid>",
      "kind": "prompt" | "pattern",
      "reason": "fabrication_trigger" | "contaminated_invariant" | "adversarial_misfired" | "reviewer_discretion",
      "persona": "<persona id, null for pattern-level>",
      "prompt": "<full prompt text, null for pattern-level>",
      "triggeringError": {
        "tool": "<tool name>",
        "errorTextKey": "<normalized error signature, lowercased, no IDs>",
        "fullError": "<original error text>"
      },
      "contextInvariant": "<probe invariant text if contamination was the reason, else null>",
      "patternRegex": "<regex pattern for pattern-level entries, else null>",
      "rejectedInRun": "<run id / timestamp>",
      "rejectedByReviewer": true,
      "markedAt": "<ISO8601>"
    }
  ]
}
```

Two kinds of entries:

- **`kind: "prompt"`** — a specific prompt by a specific persona was problematic. Fields `persona`, `prompt`, `triggeringError` are populated.
- **`kind: "pattern"`** — an error pattern (not a single prompt) was deemed unfixable. Produced when the **model-error fixer** is rejected by the reviewer — the whole pattern, not any single prompt, is marked. Fields `patternRegex` and `triggeringError.errorTextKey` are populated; `persona` and `prompt` are null.

### Anti-examples feeding generator

The prompt generator (`prompts/user-sim.md` or wherever generation lives — see Files Changed) receives a new prepended section per persona:

```markdown
## Anti-Examples — DO NOT generate semantically similar prompts

The following prompts were determined to trigger fabrication, contain contaminated
invariants, or otherwise mislead the test loop in previous runs. Avoid producing
prompts that would exercise the same behavior or produce the same errors.

{for each entry in failing-prompts for this persona, render:}
- Prompt: "{entry.prompt}"
  Reason: {entry.reason}
  Error it produced: {entry.triggeringError.fullError | truncate 200}
```

Pattern-level entries (kind: "pattern") are passed to the **model-error fixer** input rather than the generator, as a filter on which error groups it should attempt to process.

### Model-error fixer filtering

The model-error fixer currently receives all model-category errors as a pattern-matching exercise. With pattern-level failing entries in place, it filters its input:

```js
const failingPatterns = loadFailingPrompts(config)
  .entries
  .filter(e => e.kind === 'pattern');

const filteredModelErrors = modelErrors.filter(me => {
  const errorText = me.errors.map(e => e.error || '').join(' ').toLowerCase();
  return !failingPatterns.some(fp => new RegExp(fp.patternRegex, 'i').test(errorText));
});
```

If all errors match failing patterns, the model-error fixer is skipped entirely.

### Adversarial prompt support

Prompts entries (both in `prompt-set.json` for golden and in the run-transient generated sets) gain an optional field:

```json
{
  "prompt": "...",
  "adversarial": true,
  ...
}
```

Semantics:

- **Generator** may optionally produce adversarial prompts (e.g., 10% of output per persona, configurable via `adversarialRatio` in `evolve.config.mjs`, default 0). For V1 the generator does not produce adversarial prompts automatically — existing or manually-added adversarial prompts are respected, and the ratio defaults to 0 so no behavioral change is forced.
- **Grader** — when `adversarial: true` and the answerer's response contains errors, score as `pass` (expected failure), not fail.
- **Fixer input** — adversarial prompt errors are filtered out of fixer input entirely. The fixer never sees them.
- **Reviewer case-matrix** — if a fix attempt is somehow triggered by adversarial errors (shouldn't happen given the fixer filter above, but defensive), the reviewer automatically rejects the fix.

### Rejection semantics in scoring

When the reviewer drops a prompt:

1. The prompt's entry in the current run's scored-prompts list gets `invalid: true` and `invalidReason: <reviewer reason>`.
2. Score aggregation excludes `invalid` prompts from all buckets (`all`, `train`, `holdout`, `golden`).
3. The prompt's errors are NOT included in any downstream metrics.
4. An entry is added to `.mcp-evolve/failing-prompts.json` via `addFailingEntry()`.
5. The current run log's new `reviewer` section lists the dropped prompt for user visibility.

When the reviewer rejects a fix:

1. The fixer branch is NOT merged. The worktree is cleaned up without applying the diff.
2. The rejection is logged in the `reviewer` section of the run log with evidence.
3. The original errors that triggered this fix remain in the run (they are failures, not fabricated fixes) — UNLESS the prompts contributing them are ALSO dropped.

## Verification

1. **Unit tests for `lib/failing-prompts.mjs`** — load, save, add (both kinds), query by persona, query patterns, clear-all.
2. **Synthetic fabrication test** — craft a fixer diff that re-adds an `occupiedTableError` guard; run the upgraded reviewer against it; verify it outputs `decision: reject` with backend-check evidence; verify branch is NOT merged.
3. **Synthetic legitimate test** — craft a fixer diff that improves a tool description (adds a language hint); run reviewer; verify `decision: merge`.
4. **Synthetic contaminated-invariant test** — craft a run where a probe invariant violation triggers a fix attempt; reviewer should output `PROMPT_REVIEW` with `decision: drop` for the contaminated prompt, and the prompt should land in `failing-prompts.json`.
5. **Anti-example regeneration test** — add an entry to `failing-prompts.json`, run the generator, verify the new prompts do not repeat the failing pattern (manual inspection on a few samples; the test is less about exact string equality and more about the generator-LLM observing the context).
6. **Adversarial prompt test** — add a prompt with `adversarial: true` that is designed to produce an error (e.g. "cancel a transaction that doesn't exist"); verify grader scores it as pass despite the error; verify fixer doesn't see the error.
7. **Model-error-fixer pattern filter test** — add a pattern-level failing entry matching "already occupied"; synthesize model errors containing that text; verify the model-error fixer's input filtering excludes them.
8. **End-to-end replay of Round 9 incident** — run one iteration of pubman with the upgraded reviewer active on a branch where the fabricated guard was present; verify the guard gets rejected at merge time; verify the subsequent run is clean.

## Open questions / future work

- Should reviewer evidence be stored alongside rejected fixes in a reject-log, or only in the run log? (V1: run log only.)
- How to handle reviewer itself fabricating (reviewer LLM has same pattern-match risks)? (V2: self-audit loop, or human sample review.)
- Embedding-based deduplication of failing prompts (V2).
- Auto-expiry of failing entries when their `triggeringError.errorTextKey` no longer appears anywhere in the MCP source (V2).
- Shared checklist fragment between reviewer and promoter (cross-spec concern; defer until both are implemented).
- Kill-switch UX: `reviewerAuditEnabled: false` config option to revert to old reviewer behavior for emergencies.
